# 桌面伴学机器人专注度监督方案

## 1. 目标定位

本项目的核心目标不是让小程序承担算法，而是为“小智”桌面伴学机器人补充一个外部视觉感知通道。

推荐架构：

```text
废旧安卓手机
  只作为网络摄像头
        ↓
电脑端家长工作台
  拉取视频流、运行 focus_lab、展示状态、执行提醒策略
        ↓
小智 MCP / 控制桥接层
  调用小智机器人提醒能力
        ↓
小智机器人
  语音提醒、动作提醒、播放家长留言、互动鼓励
```

核心原则：

- 手机不跑专注度算法。
- 小程序不跑 MediaPipe / FaceMesh / 模型推理。
- MCP 不传输视频流，只负责最后的机器人提醒调用。
- 电脑端是视觉分析、策略判断、家长控制台和小智联动的核心。

## 2. 为什么不用小程序跑算法

小程序端运行视觉算法风险较高：

- 微信小程序运行环境受限，不适合长时间高频处理视频帧。
- 废旧手机算力、电池、发热、兼容性不可控。
- MediaPipe / FaceMesh 这类能力在小程序端落地成本高。
- 实时视频帧处理会给 JS 线程、内存和网络带来较大压力。
- 后续还要支持姿态、视线、离座、稳定性、AI 决策等功能，小程序端会越来越难维护。

因此小程序应退出核心链路，最多作为可选的状态查看端。第一版应直接做电脑端家长工作台。

## 3. 手机作为网络摄像头

### 3.1 首选方案：现成安卓应用

优先使用现成安卓网络摄像头应用，先验证完整闭环。

推荐优先级：

1. IP Webcam
   - 适合把安卓手机变成局域网 IP 摄像头。
   - 通常可提供 MJPEG 视频流或 HTTP snapshot。
   - 电脑端 OpenCV / FastAPI 可以直接拉取。

2. DroidCam
   - 更偏把手机变成电脑虚拟摄像头。
   - 适合视频会议或 OBS，也可以作为备选输入源。

3. Iriun Webcam
   - 也偏虚拟摄像头路线。
   - 通常需要电脑端驱动，工程上不如直接拉网络流清晰。

建议第一轮先使用 IP Webcam。

目标是拿到类似下面的视频地址：

```text
http://手机局域网IP:端口/video
http://手机局域网IP:端口/shot.jpg
rtsp://手机局域网IP:端口/...
```

### 3.2 什么时候才写安卓 App

只有在现成应用无法满足以下需求时，才考虑自研安卓软件：

- 自动开机启动摄像头服务。
- 固定输出 RTSP / MJPEG / snapshot。
- 禁止孩子操作手机。
- 自定义码率、分辨率、帧率。
- 局域网发现或绑定电脑端。
- 后续需要设备管理、心跳、远程配置。

自研安卓 App 不是第一阶段目标。第一阶段重点是验证：

```text
手机画面 → 电脑识别 → 小智提醒
```

## 4. 电脑端家长工作台

电脑端工作台替代当前小程序的作用，建议使用本项目已有的 Python 后端作为基础。

现有基础：

- `backend/main.py`
  - 已有 FastAPI 服务。
  - 已有 `/ws`、`/snapshot.jpg`、`/video_feed`、`/control`、`/sources` 等接口。

- `backend/engine.py`
  - 已有本地摄像头、网络摄像头、mock fallback 等边缘节点逻辑。

- `focus_lab/`
  - 已有 MediaPipe / OpenCV / 专注度特征分析实验代码。

建议把电脑端定位为 Edge Node：

```text
FastAPI 后端
  - 管理视频源
  - 拉取手机网络摄像头
  - 调用 focus_lab 分析
  - 推送实时状态
  - 输出截图 / 视频预览
  - 判断提醒事件
  - 调用小智 MCP

Web 家长界面
  - 展示视频画面
  - 展示专注度分数
  - 展示离座、分心、姿态、视线等状态
  - 设置提醒阈值
  - 设置提醒冷却时间
  - 手动发送提醒
  - 输入家长文字留言
  - 查看历史事件
```

## 5. 专注度分析链路

电脑端从手机网络摄像头读取视频流后，交给 `focus_lab` 或后端分析模块处理。

推荐输出状态：

```json
{
  "timestamp": 1778428800000,
  "studentLabel": "学生 A",
  "status": "distracted",
  "focusScore": 42,
  "awaySeconds": 0,
  "eventText": "连续分心 12 秒，建议温和提醒。",
  "metrics": {
    "gaze": 35,
    "posture": 72,
    "stability": 60,
    "presence": 95
  }
}
```

建议状态枚举：

```text
flow
normal
distracted
away
timeout_away
```

其中：

- `flow`：高度专注。
- `normal`：正常学习。
- `distracted`：分心。
- `away`：短时离座或画面中无人。
- `timeout_away`：离座时间过长。

## 6. 提醒策略

不要只根据单帧结果提醒，应使用连续状态判断，避免误报。

基础规则建议：

```text
连续分心 >= 10 秒 → 可触发提醒
连续离座 >= 5 秒 → 可触发提醒
低头/歪坐 >= 15 秒 → 可触发姿态提醒
同类提醒冷却时间 >= 60 秒
每 10 分钟最多自动提醒 3 次
```

提醒分级建议：

```text
level_1：温和提醒
  “我们把眼睛放回书本上，再坚持 5 分钟。”

level_2：鼓励式提醒
  “刚刚有点走神啦，我们一起把注意力拉回来。”

level_3：家长留言
  播放家长预设或临时输入的文字。

level_4：暂停自动提醒，通知家长
  如果连续多次无效，不要让机器人不断打扰孩子。
```

## 7. 小智 MCP 的作用

MCP 在本项目中只负责机器人能力调用，不负责视频传输、不负责算法推理。

目标调用链：

```text
电脑端检测到不专注
        ↓
提醒策略判断允许触发
        ↓
小智 Bridge 调用 MCP 工具
        ↓
小智机器人执行语音 / 动作 / 表情 / 播放留言
```

电脑端需要新增一个小智桥接模块：

```text
xiaozhi_bridge/
  mcp_client.py
  reminder_policy.py
  reminder_templates.py
  event_log.py
```

职责：

- 连接小智 MCP 服务或小智后台控制入口。
- 获取小智设备支持的工具列表。
- 调用语音提醒、动作提醒、表情显示等工具。
- 支持手动提醒和自动提醒。
- 处理失败重试、超时和日志记录。

注意：

实际工具名取决于小智设备侧注册了哪些 MCP tools。需要确认小智侧是否已经支持类似：

```text
say_text
play_audio
set_expression
perform_action
```

如果没有，需要在小智侧新增对应工具。

## 8. 家长端扩展能力

### 8.1 手动提醒

家长端页面应支持：

- 一键温和提醒。
- 一键姿态提醒。
- 一键休息提醒。
- 输入一段文字，让小智读给孩子。
- 选择提醒风格：温和、鼓励、严肃、游戏化。

建议请求格式：

```json
{
  "type": "custom_message",
  "text": "宝贝，妈妈看到你有点走神啦，我们再坚持十分钟。",
  "style": "warm",
  "issuedBy": "parent"
}
```

### 8.2 自动提醒

自动提醒应默认保守：

- 必须满足连续异常时长。
- 必须满足冷却时间。
- 必须记录每次提醒原因。
- 必须可以在界面关闭。

## 9. DeepSeek API 扩展

DeepSeek 不应处理视频帧，也不应替代基础规则引擎。

推荐分工：

```text
规则引擎
  负责硬触发：连续分心、离座、姿态异常、提醒冷却。

DeepSeek
  负责软决策：选择提醒话术、判断提醒风格、生成家长端解释。
```

只向 DeepSeek 发送结构化数据：

```json
{
  "studentAge": 8,
  "sessionMinutes": 25,
  "focusScore": 42,
  "status": "distracted",
  "awaySeconds": 0,
  "metrics": {
    "gaze": 35,
    "posture": 72,
    "stability": 60,
    "presence": 95
  },
  "recentEvents": [
    "look_away_12s",
    "posture_recovered",
    "focus_dropped"
  ],
  "lastReminderSecondsAgo": 180
}
```

DeepSeek 输出建议：

```json
{
  "shouldRemind": true,
  "reminderLevel": "level_1",
  "message": "我们把眼睛放回书本上，再坚持 5 分钟。",
  "reason": "孩子连续分心超过 10 秒，但近期提醒次数不多，适合温和提醒。"
}
```

安全原则：

- 不上传孩子视频帧。
- 不上传人脸图片。
- 不让 AI 绕过冷却规则频繁提醒。
- 家长端保留“AI 自动提醒开关”。
- 默认先做 AI 建议，再逐步开放 AI 自动触发。

## 10. 推荐开发顺序

### 阶段 1：验证手机摄像头输入

- 安装 IP Webcam。
- 确认电脑能访问手机视频流。
- 使用 OpenCV 读取视频地址。
- 将该地址接入 `backend` 的 network source。

目标：

```text
电脑端能看到手机摄像头画面
```

### 阶段 2：跑通专注度分析

- 将手机视频流接入 `focus_lab`。
- 输出 `focusScore`、`status`、`metrics`。
- 通过 `/ws` 推送实时状态。
- 通过 `/snapshot.jpg` 或 `/video_feed` 显示画面。

目标：

```text
电脑端能基于手机画面判断专注度
```

### 阶段 3：构建家长端 Web 页面

- 替代现有小程序 UI。
- 显示实时画面、专注度分数、事件列表。
- 支持视频源配置。
- 支持手动提醒按钮。

目标：

```text
家长可以在电脑上监控和操作
```

### 阶段 4：接入小智 MCP

- 新增 `xiaozhi_bridge`。
- 先实现手动按钮触发小智提醒。
- 确认小智侧 tools/list 和 tools/call 可用。
- 记录每次调用结果。

目标：

```text
家长端点按钮，小智能提醒孩子
```

### 阶段 5：自动提醒

- 新增提醒策略模块。
- 连续分心、离座、姿态异常才触发。
- 加入冷却时间和频率限制。
- 自动调用小智提醒。

目标：

```text
孩子不专注时，小智能自动且不过度地提醒
```

### 阶段 6：DeepSeek 决策辅助

- 接入 DeepSeek API。
- 根据结构化专注度数据生成提醒话术。
- 先展示 AI 建议，由家长确认。
- 再开启可配置的自动提醒。

目标：

```text
AI 帮助选择更合适的提醒方式和话术
```

## 11. 第一版最小闭环

第一版不要做太大。建议最小闭环为：

```text
IP Webcam 手机视频流
        ↓
电脑 FastAPI 拉流
        ↓
focus_lab 输出 distracted / normal
        ↓
电脑页面显示状态
        ↓
家长手动点击提醒
        ↓
小智 MCP 执行一句语音提醒
```

这个闭环跑通后，再逐步加自动提醒、家长留言、DeepSeek 决策。

## 12. 关键风险

### 视频流稳定性

废旧手机可能发热、断流、网络不稳。需要后端支持断线重连和 fallback。

### 识别误报

专注度不是单帧判断，必须用时间窗口。不要一低分就提醒。

### 小智 MCP 能力边界

需要确认小智当前支持哪些 MCP 工具。如果没有语音提醒工具，需要先在小智侧注册。

### 隐私与安全

不要上传孩子视频到云端。DeepSeek 只接收结构化状态，不接收图像。

### 过度提醒

提醒太频繁会破坏学习状态。必须有冷却时间、频率上限和家长关闭开关。

## 13. 总结

本项目应从“小程序专注度识别”转向“桌面端边缘 AI 家长工作台”。

最终定位：

```text
手机 = 网络摄像头
电脑 = 视觉分析和决策中枢
小智 = 陪伴式提醒执行端
DeepSeek = 话术和策略辅助
```

优先做最小闭环，不急于自研安卓 App，也不急于开放 AI 全自动提醒。
